# TODO 章节占位与正文草稿编写约定

本文件用于约束第 `39` 章及之后的“占位章节”与“正文草稿章节”该如何被阅读、审计和继续编写。

目标不是提供教程正文，而是让人类作者和 AI agent 在看到这些文稿时，能对它们的状态、约束和后续动作形成一致理解。

## 适用范围

- 适用于所有明确标记为“占位”“TODO-only”“规划态”“正文草稿态”的章节文稿。
- 当前至少适用于第 `39-46` 章。

## 这些文稿当前代表什么

- 它们可能处于两种状态之一：
  - **章节规格草案**
  - **正文草稿**
- 它们定义的是：
  - 章节边界
  - 正文结构或正文草稿
  - demo 规划
  - 练习规划
  - 审计检查点
- 它们**不代表**：
- 正文已经过最终审计
- 示例工程已经创建
- 练习和答案已经创建
- `teaching-api` 已经完成对应改造

## 顶部“占位元信息”的含义

每个占位章节顶部都应包含一段 `yaml` 代码块，用于让人类和 AI 快速读取当前状态。

字段含义固定如下，且值域应保持统一，不要自由发明新枚举词：

- `chapter_type`
  - 允许值只使用：
    - `planned_placeholder`
    - `body_draft`
- `writing_status`
  - 允许值只使用：
    - `todo_outline_only`
    - `body_draft_pending_audit`
- `body_written`
  - `false` 代表正文尚未开始写；`true` 代表正文草稿已经写出。
- `demo_created`
  - `false` 代表对应 demo 尚未创建。
- `exercise_created`
  - `false` 代表对应练习 starter 尚未创建。
- `answer_created`
  - `false` 代表对应练习答案尚未创建。
- `api_changes_required`
  - 只使用布尔值 `true/false`，表示本章后续是否预期需要改动 `teaching-api`。
- `do_not_treat_as_finished_content`
  - `true` 代表任何 agent 都不能把该文稿当成已完成章节。

推荐对应关系如下：

- 纯占位稿：
  - `chapter_type: planned_placeholder`
  - `writing_status: todo_outline_only`
  - `body_written: false`
- 正文草稿：
  - `chapter_type: body_draft`
  - `writing_status: body_draft_pending_audit`
  - `body_written: true`

## 结构区块约定

### 占位稿阶段必须保留的区块

### `TODO 模块拆分`

- 这是章节正文未来的**主结构**。
- `模块 0` 到 `模块 7` 是推荐顺序，不是随机想法列表。
- 后续写正文时，优先按这些模块顺序展开。
- 如果要改模块顺序或增删模块，必须同步检查相邻章节衔接和本章审计点。

### `TODO Demo 规划`

- 这是后续 demo 的最低交付预期。
- 这里的条目不是“可选建议”，而是后续创建 demo 时应优先满足的目标。
- 如果最终 demo 无法覆盖这些点，必须回写文稿说明原因。

### `TODO 练习规划`

- 这是后续练习 starter / answer 的最低交付预期。
- 练习不应只是照抄 demo，而应有独立补全点。

### `TODO 审计检查点`

- 这是章节在被视为“完成”前必须经过的审查清单。
- 后续 agent 如果补正文、demo 或练习，应逐条对照这些检查点。
- 如果检查点本身已经不成立，应先改文稿，再继续实现。

### 正文草稿阶段必须保留的区块

当章节已经从 TODO-only 推进到“正文草稿”后，可以把 `TODO 模块拆分` 落实进正式正文结构，但仍应保留以下固定标题，方便后续作者和 AI agent 继续落地：

### `后续 Demo 规划（未实现）`

- 这是后续 demo 的最低交付预期。
- 不要把它和“本轮已完成内容”混写在一起。

### `后续练习规划（未实现）`

- 这是后续练习 starter / answer 的最低交付预期。
- 练习应保留独立补全点，不应只是照抄 demo。

### `审计检查点（待完成）`

- 这是正文草稿走向“可交付章节”前必须逐条回查的清单。
- 标题和语义尽量保持统一，避免不同章节各写各的“待补说明”“待审计点”“未完成事项”。

## 后续 agent 的默认行为约束

- 不要把 TODO 文稿误读成正式教程内容。
- 不要把“正文草稿”误读成已完成章节。
- 不要只补某个 demo，却不更新对应占位文稿。
- 不要修改章节标题、编号、文件名，除非明确要重排后续主线。
- 如果改动章节范围，必须同步更新：
  - 上一章 / 下一章链接
  - 相邻占位文稿中的衔接描述
  - 本章的审计检查点

## 推荐的后续落地顺序

1. 先从占位稿补到正文草稿
2. 再补 demo
3. 再补练习 starter
4. 再补练习答案
5. 最后按“审计检查点”回查并决定是否可交付

## 明确的非目标

- 本文件不规定具体措辞。
- 本文件不替代章节正文。
- 本文件不自动授权任何 agent 去新增 API、demo 或练习。

是否真正实施，仍应以当前任务要求为准。
