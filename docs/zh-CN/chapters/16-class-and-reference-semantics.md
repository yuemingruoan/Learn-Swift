## 为什么这一章要学 class

到目前为止，你已经接触过不少 Swift 类型：

- `Int`、`Double`、`String`
- 数组和字典
- 枚举
- `struct`

如果你一路按当前教程的节奏学下来，很容易形成一个很自然的直觉：

- 变量赋值给另一个变量后，后面应该各改各的

这个直觉在很多前面学过的例子里都没有明显问题。

例如：

- 一个 `Int` 赋值给另一个 `Int`
- 一个 `String` 赋值给另一个 `String`
- 一个 `struct` 实例赋值给另一个变量

在这些场景里，你通常不会看到“改了 B，A 也跟着变”。

但 `class` 会打破这个直觉。

这也是为什么这一章必须单独讲，而且要讲透。因为很多初学者并不是不会写 `class` 语法，而是：

- 看不懂运行结果为什么和 `struct` 不一样

所以这一章的重点不是“面向对象思想”，也不是继承、多态、设计模式，而是更基础的一件事：

- `class` 的值在程序里到底是怎样被变量引用和共享的

## class 的最基础写法

先看一个最基础的 `class` 例子：

```swift
class StudentProfile {
    var name: String
    var score: Int

    init(name: String, score: Int) {
        self.name = name
        self.score = score
    }

    func printSummary() {
        print("姓名：\(name)，分数：\(score)")
    }
}
```

你现在可以先把它和之前学过的 `struct` 对照着理解：

- `StudentProfile` 是类型名
- `name` 和 `score` 是属性
- `printSummary()` 是方法
- `init(...)` 是创建实例时会用到的初始化器

这里先不要被新术语吓住。当前阶段你只需要先抓住最重要的事实：

- `class` 也是自定义类型
- 它里面也可以放属性和方法

## class 实例怎么创建

如果要创建上面这个类的实例，最基础的写法如下：

```swift
let student = StudentProfile(name: "小林", score: 82)
student.printSummary()
```

这一段里最值得注意的是：

- `StudentProfile(name: "小林", score: 82)` 会创建一个实例
- `student` 保存的是这个实例
- 后面可以用点语法访问属性和方法

例如：

```swift
print(student.name)
student.score = 90
student.printSummary()
```

看到这里，你很可能会觉得：

- 这和 `struct` 看起来差不多

这正是这一章最容易让人掉以轻心的地方。

因为：

- `class` 和 `struct` 的表面写法确实很像
- 但它们在赋值和传参时的行为差别非常大

## 表面很像，但行为不一样

这一节是全章真正的核心。

如果你只记住一句话，那就记这一句：

- `class` 和 `struct` 最大的差别，不在于有没有属性和方法，而在于赋值之后是不是共享同一个实例

为了看清这个差别，我们先回顾 `struct`。

## 先回顾 struct 的值语义

先看下面这个结构体例子：

```swift
struct ScoreRecord {
    var score: Int
}

var a = ScoreRecord(score: 80)
var b = a
b.score = 100

print(a.score)
print(b.score)
```

当前阶段，你可以先把运行结果理解成：

- `a` 里有一份自己的值
- `b = a` 之后，`b` 得到的是另一份独立的值
- 所以后面改 `b.score`，不会影响 `a.score`

也就是说，这里体现的是一种“各管各的”行为。

这就是值语义最适合初学者的直观理解方式：

- 赋值后，后面的修改通常互不影响

## 再看 class 的引用语义

现在看一个几乎同构的类例子：

```swift
class ScoreBox {
    var score: Int

    init(score: Int) {
        self.score = score
    }
}

let a = ScoreBox(score: 80)
let b = a
b.score = 100

print(a.score)
print(b.score)
```

很多读者第一次看到这里时，会本能地猜：

- `a.score` 还是 `80`
- `b.score` 变成 `100`

但实际结果通常是：

```text
100
100
```

这不是编译器出错，也不是复制失败，而是 `class` 本来就这样工作。

因为在这里：

- `b = a` 并没有创建一个新的 `ScoreBox`
- `a` 和 `b` 都在引用同一个实例

所以通过 `b` 修改 `score` 时，实际上改的就是那一个共享实例里的数据。

## 什么叫“两个变量指向同一个实例”

你可以先用一个不那么底层、但足够好用的比喻来建立直觉：

- `struct` 更像是“复制了一份表格内容”
- `class` 更像是“给同一个盒子又贴了一个名字标签”

如果是复制表格：

- 改第二份表格
- 第一份表格不会跟着变

如果是给同一个盒子贴两个标签：

- 你通过任意一个标签打开盒子去改里面的内容
- 另一个标签看到的当然也是改过后的结果

这就是为什么 `class` 会出现：

- 改 B，A 也变了

## 这就是引用语义

当前阶段，可以先把引用语义理解成：

- 变量保存的重点不是“独立内容副本”
- 而是“对某个实例的引用”

于是：

- 赋值时，复制的往往不是实例本身
- 而是“指向这个实例”的这层关系

所以 `class` 最关键的判断标准不是：

- 这个类型里有多少属性

而是：

- 当前这几个变量是不是在操作同一个实例

## 为什么这不是 bug，而是 class 的正常语义

初学者第一次遇到 `class` 时，最容易冒出的想法是：

- 我明明只是改了 `b`
- 为什么 `a` 也变了

这时一定要主动提醒自己：

- 这里不是两个独立实例
- 而是两个变量在共享同一个实例

所以真正应该问的问题不是：

- “为什么 `a` 被偷偷改了？”

而应该是：

- “我是不是其实一直只在操作同一个实例？”

只要这个问题想明白了，后面很多现象都会顺下来。

## let 和 var 在 class 里到底限制了什么

这一点也非常容易混淆。

先看这段代码：

```swift
let student = StudentProfile(name: "小林", score: 82)
student.score = 90
```

很多人看到 `let` 会下意识以为：

- 既然是常量，那里面的分数也不该能改

但对于类实例来说，`let` 限制的重点其实是：

- 这个变量不能再改成引用别的实例

而不是：

- 这个实例内部所有 `var` 属性都不能改

也就是说：

- `let student = ...` 之后，`student` 这个名字不能再绑定到另一个实例
- 但如果当前实例里的属性本身是 `var`，通常仍然可以修改

例如：

```swift
let student = StudentProfile(name: "小林", score: 82)
student.score = 90

// student = StudentProfile(name: "小周", score: 95)
// 上面这行会报错，因为 student 不能再改为引用新的实例
```

这和很多读者对 `struct` 的直觉差异很大，所以这一点一定要单独记住。

## 把 class 传给函数时会发生什么

引用语义不只会体现在赋值上，也会体现在传参上。

先看下面这个函数：

```swift
func increaseScore(profile: StudentProfile, by delta: Int) {
    profile.score += delta
}
```

然后调用它：

```swift
let student = StudentProfile(name: "小林", score: 82)
increaseScore(profile: student, by: 5)
print(student.score)
```

这时你通常会看到：

- `student.score` 已经变成了 `87`

这说明：

- 函数里操作的不是一个完全独立的新实例
- 它操作的仍然是调用方手里的那一个实例

这也是引用语义的典型表现：

- 只要你还在操作同一个类实例，变化就会被共享看到

## `===` 和 `==` 有什么不同

这一章只先讲最基础、最需要的部分。

对于类实例，Swift 还提供了一种很重要的比较方式：

```swift
===
```

它的作用不是判断“内容像不像”，而是判断：

- 这两个变量是不是引用同一个实例

例如：

```swift
let a = StudentProfile(name: "小林", score: 82)
let b = a
let c = StudentProfile(name: "小林", score: 82)

print(a === b)
print(a === c)
```

这里通常会得到：

```text
true
false
```

原因是：

- `a` 和 `b` 引用的是同一个实例
- `a` 和 `c` 虽然内容一样，但它们是两个分别创建出来的实例

当前阶段只要先记住：

- `===` 看的是“是不是同一个实例”
- 它不等于“内容是否相同”

至于 `==`、`Equatable` 这些内容，这一章先不展开。

## 如果你学过 C++ 或 Java，这一章该怎么建立直觉

这一节只做帮助理解的对比，不展开三门语言的完整对象模型。

### 如果你学过 Java

那么 Swift 的 `class` 会比较顺手。

因为在最基础的行为上，它们都很接近：

- 变量通常不是直接装着一整份独立对象内容
- 赋值后，多个变量可能引用同一个对象或实例
- 改一个地方，另一个地方也可能看到变化

所以如果你有 Java 背景，这一章最容易理解的部分就是：

- `class` 的引用语义

### 如果你学过 C++

那么这一章最需要调整的直觉是：

- Swift 的 `class` 不是“赋值后自动得到一份独立副本”的那套直觉

因为很多 C++ 初学场景里，`class` 更容易先被理解成：

- 一种自定义类型对象
- 直接赋值时更接近复制内容

而 Swift 里的 `class` 不是往这个方向工作的。

所以如果你有 C++ 背景，这一章最该主动提醒自己的就是：

- Swift 的 `class` 行为上更接近 Java
- 而 Swift 里更像“直接复制值”的直觉，很多时候反而更适合放在 `struct` 上理解

## 什么时候优先想到 struct，什么时候想到 class

这一章不讲设计哲学，只给你一个当前阶段足够实用的判断标准。

### 更适合先想到 `struct` 的情况

- 你希望赋值后互不影响
- 你主要是在表示一组数据
- 你不需要多个位置共享同一个可变实例

### 更适合考虑 `class` 的情况

- 你明确需要多个地方一起引用同一个实例
- 你希望一个地方修改后，别的地方也能看到最新状态
- 你要表达的重点不只是“数据长什么样”，还包括“当前是不是同一个实例”

如果你现在还不确定该怎么选，也没关系。当前阶段最稳妥的做法是：

- 先根据“是否需要共享同一个实例”来判断

## 本章最需要建立的判断标准

这一章最重要的不是记住多少个术语，而是建立下面这条判断习惯：

- 遇到 `class` 时，先问自己：当前这些变量是不是在操作同一个实例？

如果答案是“是”，那么：

- 一个地方的修改会被其他引用一起看到

如果答案是“不是”，那么：

- 它们才是真正彼此独立的两份东西

## 常见错误

### 1. 以为 `class` 和 `struct` 赋值后的行为一样

这是最常见的误区。

如果你默认以为：

- `b = a` 之后，`b` 一定是一份独立副本

那你几乎一定会在 `class` 这里看不懂输出结果。

### 2. 把“内容一样”误以为“同一个实例”

例如：

- 两个学生实例姓名和分数一样

这并不等于：

- 它们就是同一个实例

这也是为什么 `===` 很重要。

### 3. 误解 `let`

很多读者会误以为：

- 只要变量前面用了 `let`
- 那类实例内部属性也绝对不能改

但对类实例来说，`let` 限制的是：

- 变量绑定不能换到别的实例

### 4. 看到外部状态变化就以为函数“偷偷返回了东西”

如果你把类实例传进函数，函数里改了属性，外部也会看到变化。

这不是因为函数暗中“回传”了整个对象，而是因为：

- 函数和外部一直在操作同一个实例

## 本章小结

这一章最需要记住的不是零散语法，而是下面这组关系：

- `class` 也是一种自定义类型
- `class` 里也可以有属性、方法和初始化器
- `class` 和 `struct` 最核心的差别在于赋值和传参时的语义
- `struct` 更容易体现值语义
- `class` 更容易体现引用语义
- `===` 用来判断两个变量是不是引用同一个实例

如果你现在已经能稳定判断：

- 当前这几个变量是不是在共享同一个实例

那么这一章最重要的目标就已经完成了。
