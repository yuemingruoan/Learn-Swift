## 先看本章最常见的通用代码外形

这一章没有像 `actor`、`async let` 那样特别新的关键字，所以更适合先看“共享状态问题最常见会长成什么样”。

当前阶段最值得先记住的，通常是下面三种代码外形。

### 1. 多个任务同时碰同一个共享对象

```swift
final class StudyProgress {
    var finishedCount = 0
}

let progress = StudyProgress()

Task {
    progress.finishedCount += 1
}

Task {
    progress.finishedCount += 1
}
```

这类写法最值得你先认出来的是：

- 两个任务都拿到了同一个引用类型实例
- 它们都能直接修改同一份可变状态

### 2. 先读旧值，再写新值

```swift
let oldValue = progress.finishedCount
let newValue = oldValue + 1
progress.finishedCount = newValue
```

这类写法单线程里很普通，但放进并发里时就要立刻提高警惕，因为它依赖：

- 先读到的旧值
- 写回时这份旧值仍然有效

### 3. 先检查条件，再根据条件修改

```swift
if workshop.seatsLeft > 0 {
    workshop.seatsLeft -= 1
}
```

这类写法也很常见，但在并发里它和前一种其实是一家人：

- 都依赖“我刚刚看到的状态”
- 都假设“我接下来修改时，这个状态还没被别人改掉”

如果你先把这三种外形认熟了，后面读本章时就更容易知道自己到底在防什么。

## 为什么在学 `async/await` 之后马上教共享状态

上一章里，你已经学会了：

- 异步函数会在中途等待
- 多个独立工作可以先启动，再等待结果
- `Task` 可以把异步工作发出去

但这还不是并发真正难的地方。

并发最容易让初学者真正踩坑的，往往不是：

- `await` 写在哪
- `Task` 该不该写

而是：

- **如果两个任务同时碰到同一份可变数据，会发生什么**

例如：

- 两个任务同时给“今日完成任务数”加一
- 两个任务同时往“学习日志数组”里追加内容
- 两个任务同时检查“当前是否还有空余名额”，然后都决定写入

这时你会遇到一种非常典型的并发困惑：

- 每一行代码单独看都很合理
- 但最终结果有时对，有时不对

这正是共享状态问题开始出现的地方。

## 先说结论：并发真正难的不是“同时运行”，而是“同时改同一份状态”

很多人刚学并发时，会把注意力全部放在：

- 能不能并行
- 有没有更快
- 到底是不是多线程

这些问题不是完全不重要，但当前阶段更值得先抓住的，是下面这条判断：

- **多个任务同时存在，不一定危险**
- **多个任务同时访问同一份可变状态，才是真正容易出问题的地方**

例如：

- 一个任务加载今日任务列表
- 一个任务加载提醒文本

如果它们彼此独立，只是同时推进，通常没有什么共享状态问题。

但如果两个任务都要去改：

- `finishedCount`
- `currentUser`
- `draftNotes`
- `cachedDashboard`

那就不一样了。

因为这里已经不只是“同时做事”，而是：

- **同时碰同一份会变化的数据**

## 什么叫共享状态

当前阶段可以先把“共享状态”近似理解成：

- 有多段代码都能读它
- 并且至少有一段代码能改它

例如：

```swift
final class StudyProgress {
    var finishedCount = 0
}
```

如果这个 `StudyProgress` 实例被多个任务同时持有，那么：

- 每个任务都可能读取 `finishedCount`
- 每个任务也都可能修改 `finishedCount`

这时 `finishedCount` 就构成了共享状态。

这里要特别注意，“共享状态”不等于“全局变量”。

它也可能是：

- 某个对象的属性
- 某个数组或字典
- 某个缓存对象
- 某个被多个闭包一起捕获的变量

只要它被多方共同访问，并且其中有人能改，它就已经进入了并发风险区。

## 为什么“读一下再改一下”会变得危险

很多共享状态问题，表面上看都很普通。

例如你只是想做一件非常自然的事：

```swift
progress.finishedCount += 1
```

这行代码看起来很简单，但它背后通常不是一个不可再分的“神奇动作”。你当前阶段可以先把它近似理解成三步：

1. 先读取旧值
2. 再计算新值
3. 最后写回去

如果只有一个任务这么做，通常没有问题。

但如果两个任务几乎同时这么做，就可能出现：

1. 任务 A 读到旧值 `0`
2. 任务 B 也读到旧值 `0`
3. 任务 A 写回 `1`
4. 任务 B 也写回 `1`

最后结果是：

- 你原本以为加了两次
- 实际却只得到一次

这类问题最难受的地方在于：

- 代码没有明显崩掉
- 编译器也不一定当场指出来
- 输出只是“偶尔不对”

而这正是并发中的典型危险信号。

## 一个最基础的例子：完成数为什么会丢失

先看一个教学化的例子：

```swift
final class StudyCounter {
    var finishedCount = 0

    func markFinished() {
        let oldValue = finishedCount
        let newValue = oldValue + 1
        finishedCount = newValue
    }
}
```

单看 `markFinished()`，这段代码完全正常。

但如果两个任务同时调用它，你就要开始警惕：

- 它们会不会读到同一个旧值
- 它们会不会把彼此的写入覆盖掉

问题并不在于“加一”这个业务，而在于：

- 这段逻辑依赖旧状态
- 然后又把新状态写回原位置

这种“先读旧值，再写新值”的模式，在并发里非常常见，也非常容易出问题。

## 什么叫竞态条件

当前阶段可以先把“竞态条件”近似理解成：

- **多个执行路径都和同一份状态有关**
- **最终结果取决于谁先发生、谁后发生**

也就是说，程序不是只看“写了什么”，还开始看：

- 先后顺序刚好怎样交错

这就是“竞态”两个字最值得先抓住的地方。

例如下面这类结果就很像竞态条件：

- 有时候输出 `2`
- 有时候输出 `1`
- 有时候日志顺序不同，但代码完全没改

当你遇到这种“结果依赖时机”的代码，就应该条件反射地想到：

- 这里是不是有共享状态
- 这里是不是有竞态条件

## 数组和字典也不天然安全

很多初学者第一次意识到共享状态时，脑子里容易只想到：

- 一个 `Int`
- 一个计数器
- 一个 `Bool`

但实际项目里更常见的，往往是数组和字典。

例如：

```swift
final class StudyLogStore {
    var logs: [String] = []
}
```

如果多个任务都去做：

```swift
store.logs.append("完成了第 30 章")
```

那它们碰到的就不再只是一个数字，而是一整份可变集合。

字典也一样：

```swift
final class DashboardCache {
    var values: [String: String] = [:]
}
```

如果多个任务同时读写同一个键值缓存，同样会进入共享状态问题。

所以当前阶段请不要把问题缩得太小，以为只有“自增计数器”才算并发风险。真正应该记住的是：

- **任何会变化的共享集合，也都是共享状态**

## 值类型并不等于“放进并发里就自动安全”

这是一个特别常见的误区。

Swift 的 `struct` 很重要，值语义也很重要。但这不等于：

- 只要我用了结构体，所有并发问题就自动消失

例如：

```swift
struct StudyTask {
    var title: String
    var isFinished: Bool
}
```

`StudyTask` 本身是值类型，这没有问题。

但如果你把很多 `StudyTask` 放进一个共享数组里，再让多个任务一起修改这个数组，那么真正的问题就变成了：

- 那个被共享出去的可变容器是谁

也就是说，当前阶段不要只盯住“元素是不是 struct”，还要追问：

- 这份可变状态最终被谁共享了出去

## 闭包捕获也可能把共享状态带进来

前面你已经学过闭包会捕获外部变量。

这件事放到并发里时，要开始多想一步：

- 如果多个异步任务都捕获并修改同一个外部变量，会怎样

例如：

```swift
var messages: [String] = []

let taskA = Task {
    messages.append("A")
}

let taskB = Task {
    messages.append("B")
}
```

当前阶段你不需要立刻追底层实现，但应该先建立这个警惕：

- 这些任务看起来只是各自追加一条消息
- 实际上它们都在碰同一个 `messages`

所以问题的核心仍然不是“闭包”本身，而是：

- 闭包把共享状态一起带进了并发执行路径

## 一个更接近业务的例子：名额判断为什么会失效

共享状态问题并不总是“丢一个加一”这么直观。

更常见的一类业务错误是：

- 先检查条件
- 再根据条件写入

例如：

```swift
final class WorkshopCenter {
    var seatsLeft = 1

    func register(name: String) -> Bool {
        if seatsLeft <= 0 {
            return false
        }

        seatsLeft -= 1
        return true
    }
}
```

单看这段逻辑也很像没问题：

- 还有名额就报名成功
- 没名额就失败

但如果两个任务几乎同时调用 `register(name:)`，你就会发现：

- 它们都可能在“扣减前”看到还有 `1` 个名额
- 最后却都返回成功

这类问题比“加一丢失”更值得警惕，因为它直接会把业务规则打穿。

也就是说，并发中的共享状态问题，很多时候并不是：

- 数据格式坏了

而是：

- **业务约束失效了**

## 当前阶段最实用的分析顺序

当你怀疑一段并发代码有问题时，先不要急着背关键词。更稳妥的顺序通常是：

1. 先找出哪份状态会变
2. 再看这份状态是不是会被多个任务共同访问
3. 再判断这些访问里，是否存在“读旧值再写新值”或“先检查再修改”这种模式
4. 最后再决定应该怎么隔离或重构

这个顺序非常重要，因为很多初学者一上来会反过来：

- 先问“该用哪个语法修”

但如果你连哪份状态在共享都没找准，就很容易修偏。

## 为什么这一章先不急着讲“怎么修”

你可能会问：

- 既然已经知道共享状态危险，为什么这一章不立刻上最终解法

原因很简单：

- 如果你还没把“问题本身”看清楚，后面的解法很容易被你误解成魔法

例如下一章会讲 `actor`。如果你现在还没建立共享状态直觉，就很容易把它误解成：

- 新版 class
- 自动更快的并发容器
- 只要用了就万事大吉

但它真正要解决的，其实就是：

- **如何把共享可变状态关进明确的隔离边界**

所以这一章刻意先把“哪里危险”讲清楚，而不是急着把语法名词堆上来。

## 什么时候共享状态其实不是问题

这一点也要补一句，避免把读者吓成“只要有并发就全都危险”。

下面几种情况，通常就没有本章说的核心问题：

- 多个任务只是各算各的局部结果，最后再汇总不可变结果
- 多个任务同时读取同一份不可变数据
- 每个任务各自持有自己那份独立副本

例如：

- 一个任务计算总时长
- 一个任务生成提醒文本

如果它们只是最后把结果交给调用者合并，而不是一起改同一个共享变量，那么问题就会小很多。

所以并发分析最重要的，不是“看到多个任务就紧张”，而是：

- **看到共享可变状态才提高警惕**

## 常见误区

### 1. 以为只要用了并发，就一定会有共享状态问题

不是。

多个任务彼此独立时，并不会自动产生本章的问题。

### 2. 以为只有全局变量才算共享状态

不是。

对象属性、数组、字典、闭包捕获变量都可能成为共享状态。

### 3. 以为 `struct` 天生就能解决所有并发风险

不是。

关键不只在于类型是不是值类型，还在于可变状态是否被共享出去。

### 4. 以为“偶尔结果不对”只是日志顺序问题

不是。

日志顺序变化有时只是并发表象，真正的问题可能是业务状态被覆盖了。

### 5. 以为只有稳定复现的问题才能说明有竞态

不是。

很多竞态的典型特征恰恰是：

- 不稳定
- 难稳定复现
- 但一旦出现，就说明边界有问题

## 本章小结

这一章最需要记住的是下面这组关系：

- 并发真正危险的地方，往往不是“同时运行”，而是“同时改同一份可变状态”
- 共享状态指的是：多方都能访问，而且至少一方能修改的数据
- `+= 1`、`append`、字典写入、先检查再修改，都是共享状态里要重点警惕的模式
- 竞态条件的核心特征是：最终结果依赖执行时机
- `struct`、数组、字典、闭包捕获变量都可能牵涉共享状态问题
- 当前阶段最重要的分析顺序是：先找共享状态，再谈隔离方案

如果你现在已经开始能稳定地问出下面这些问题：

- 这份状态是谁在改
- 会不会有两个任务同时碰它
- 这里是不是“先读旧值，再写新值”

那么这一章最重要的目标就已经达到了。
