## 为什么在 URLSession 之前先补 Web 基础

很多初学者第一次写网络请求时，最容易掉进一种状态：

- 会照着示例把请求发出去
- 会照着示例把 JSON 解出来
- 但一遇到报错、状态码、登录态、Header，就开始只剩名词，不知道它们分别在解决什么问题

例如下面这些词，后面几乎一定会出现：

- URL
- HTTP
- GET / POST
- Header
- Status Code
- Cookie
- Token
- Authorization

如果不先把这些词的含义理清，后面遇到 `URLSession` 相关代码时，很容易变成：

- 只是背调用方式
- 不知道为什么这里要带 Header
- 不知道为什么这里明明拿到了响应，却还算失败
- 不知道为什么登录一次后，后续请求还能继续识别用户

所以这一章的任务不是教你“怎么写代码”，而是先把下面这个问题讲清楚：

- **当一个 Swift 程序去和远程服务通信时，中间到底发生了什么**

## 先看全景图：客户端、服务器、接口、资源

当前阶段可以先把常见角色理解成下面这样：

- **客户端（Client）**：发起请求的一方，例如你的 iPhone App、浏览器、桌面程序
- **服务器（Server）**：接收请求并返回结果的一方
- **接口（API）**：服务器暴露出来、允许客户端调用的一组访问入口
- **资源（Resource）**：你真正想获取或操作的东西，例如用户资料、任务列表、文章详情

你可以先把它们的关系近似理解成：

```text
客户端 --按照接口规则发请求--> 服务器
客户端 <--收到响应结果--------- 服务器
```

如果再具体一点，可以把它想成：

```text
学习 App -> 请求“今日任务列表” -> 服务器
学习 App <- 返回任务 JSON ----- 服务器
```
我们首先需要建立一个概念：

- App 并不是“直接拿到一个 Swift 数组”
- 它是先通过网络去请求一个远程资源
- 服务器再把结果按某种格式返回给你

而这个“返回格式”，很多时候就是 JSON。

## 什么是 URL：资源地址长什么样

后面你写网络请求时，经常会先看到一个 URL：

```text
https://api.example.com/v1/tasks?finished=false&page=1
```

不要把它只当成“一长串字符串”。它其实是在定位一个网络资源。

可以先这样拆：

- `https`：协议（scheme）
- `api.example.com`：域名（host）
- `/v1/tasks`：路径（path）
- `finished=false&page=1`：查询参数（query）

你可以把这几个部分先近似理解成：

- 协议：我要通过什么通信规则访问
- 域名：我要去找哪台服务器
- 路径：我想访问这台服务器上的哪类资源
- 查询参数：我还想补充哪些筛选条件

例如：

```text
https://api.example.com/v1/tasks?finished=false&page=1
```

它可以先读成：

- 用 `https`
- 访问 `api.example.com`
- 找 `/v1/tasks` 这条资源路径
- 并附带 `finished=false` 和 `page=1` 这两个查询条件

当前阶段不必细究 `DNS`、`TCP`、`TLS` 等细节，只要先记住：

- **URL 解决的是“请求发到哪里”**

## 一次 HTTP 请求到底发生了什么

当客户端去请求一个接口时，最粗略的过程通常可以先理解成下面这几步：

1. 客户端准备一份请求
2. 请求里写清楚要访问哪个 URL、用什么方法、要不要带额外信息
3. 服务器收到后开始处理
4. 服务器返回一份响应
5. 客户端根据响应状态和响应内容决定下一步

可以先把它画成：

```text
客户端
  -> 发请求（URL + 方法 + Header + 可选 Body）
服务器
  -> 处理请求
服务器
  -> 返回响应（状态码 + Header + Body）
客户端
  -> 判断成功/失败，再决定是否解析 JSON
```

这条顺序很重要，因为后面很多问题其实都发生在不同层：

- 连服务器都没连上：这是通信层问题
- 服务器回了 `404`：这是 HTTP 响应状态问题
- 服务器回了 JSON，但字段不匹配：这是解码问题

如果你把这三件事混成一件事，后面排查网络请求时会很吃力。

## HTTP 请求由什么组成

当前阶段你只需要先认识请求里最常见的几部分。

### 1. 请求方法（Method）

最常见的方法有：

- `GET`：获取数据
- `POST`：提交新数据
- `PUT`：整体更新
- `PATCH`：部分更新
- `DELETE`：删除资源

你可以先这么认为：

- `GET` 更像“我要获取数据”
- `POST` 更像“我要提交一份新数据”

后面写 App 网络请求时，最早和最多遇到的通常也是这两类。

### `GET` 和 `POST` 的参数通常放在哪里

这也是初学者非常容易混淆的一点。

先说一个足够实用的结论：

- `GET` 请求的参数，通常放在 URL 里
- `POST` 请求的参数，通常放在请求体里

先看 `GET` 参数最常见的书写格式：

```text
https://api.example.com/v1/tasks?finished=false&page=1&keyword=swift
```

这一段可以拆成：

- `?`：表示查询参数从这里开始
- `finished=false`：一个参数，含义是 `key=value`
- `&`：表示“下一个参数继续接在后面”
- `page=1`：第二个参数
- `keyword=swift`：第三个参数

也就是说，`GET` 请求里最常见的查询参数格式就是：

```text
?key1=value1&key2=value2&key3=value3
```

你可以先把它理解成：

- `?` 后面开始写参数
- 每个参数都长得像 `参数名=参数值`
- 多个参数之间用 `&` 连接

例如：

```text
/v1/tasks?finished=false&page=1
```

可以读成：

- 请求 `/v1/tasks`
- 并附带两个参数
- `finished=false`
- `page=1`

如果只有一个参数，就不需要 `&`，例如：

```text
/v1/tasks?page=1
```

当前阶段先在心里留个印象就够了：只要后面一看到 URL 里有 `?` 和 `&`，就能意识到“这里是在传递查询参数”就足够了。

例如下面这个 `GET`：

```text
GET /v1/tasks?finished=false&page=1 HTTP/1.1
Host: api.example.com
```

这里的：

- `finished=false`
- `page=1`

都属于查询参数，也就是附在 URL 后面的参数。

你可以把它先理解成：

- 我要请求任务列表
- 并顺便告诉服务器“只看未完成任务”“当前看第 1 页”

而 `POST` 更常见的情况是把参数放在请求体里，例如：

```text
POST /v1/tasks HTTP/1.1
Host: api.example.com
Content-Type: application/json

{
  "title": "复习闭包",
  "estimatedHours": 2
}
```

这里真正提交的数据不在 URL 里，而是在 Body 里。

你可以先把这两种方式近似理解成：

- `GET`：更像“我带着筛选条件去拿数据”
- `POST`：更像“我把一份数据提交给服务器”

### 为什么会有这种差别

因为这两类请求最常见的用途本来就不一样：

- `GET` 通常用于获取资源
- `POST` 通常用于提交新数据或触发某个操作

所以很多接口在设计时，会自然形成下面这种习惯：

- 获取列表、搜索、筛选、分页：常放在 URL 查询参数里
- 登录、创建、提交表单、上传结构化数据：常放在请求体里

例如：

- `/v1/tasks?finished=false&page=1`
- `/v1/articles?keyword=swift&sort=latest`

看起来都更像“我要什么范围的数据”。

而像：

- 登录信息
- 新建任务内容
- 修改用户资料

这些则更像“我要交给服务器处理的一份数据”。

### 这里要注意什么

当前阶段先记住下面三个边界就够了：

- 这是一种非常常见的实践，不是死板到绝对不能变的规则
- `GET` 的参数常见于 URL，所以你阅读接口文档时往往先看查询参数
- `POST` 的参数常见于请求体，所以你阅读接口文档时要特别看 `Body` 的数据格式

也就是说，后面当你真正开始写客户端请求时，不要只问：

- “这个接口要什么参数”

还要继续追问：

- “这些参数是放在 URL 里，还是放在请求体里”

### 2. 请求头（Header）

请求头可以理解成：

- **这次请求附带的说明信息**

例如它可能会说明：

- 我发送的内容是什么格式
- 我希望你返回什么格式
- 我是谁，或者我带着什么凭证来的

例如后面很常见的一类 Header 是：

```text
Authorization: Bearer <token>
```

它表达的不是“真正业务数据”，而是：

- 这次请求附带了一份身份凭证

### 3. 请求体（Body）

请求体通常出现在需要提交数据的场景，例如：

- 提交登录信息
- 新建一条任务
- 修改用户资料

例如：

```json
{
  "title": "复习闭包",
  "estimatedHours": 2
}
```

这类 JSON 如果放在请求里，它表达的是：

- **我要发给服务器的数据内容**

### 一个最小请求示意

```text
POST /v1/login HTTP/1.1
Host: api.example.com
Content-Type: application/json

{
  "email": "alice@example.com",
  "password": "123456"
}
```

当前阶段不需要把语法背下来，你只需先看清它表达了什么：

- 我要访问 `/v1/login`
- 我用的是 `POST`
- 我发的是 JSON
- 我在 Body 里提交了登录信息

## HTTP 响应由什么组成

服务器返回的响应，最常见也可以先拆成三部分。

### 1. 状态码（Status Code）

状态码用来表达：

- **服务器如何看待这次请求**

它回答的不是“你的业务是否绝对成功”，而是更接近：

- 这次请求在 HTTP 这层看起来结果怎样

### 常见状态码

当前阶段不需要死记硬背每个状态码对应的含义，但下面这些常见的状态码需要留个印象：

- `200 OK`：请求成功，服务器正常返回结果
- `201 Created`：创建成功，常见于新建资源后
- `400 Bad Request`：请求本身有问题，例如参数不合法
- `401 Unauthorized`：身份校验没有通过，常见于没登录、凭证无效、凭证过期
- `403 Forbidden`：服务器知道你是谁，但你没有权限做这件事
- `404 Not Found`：请求的资源不存在，或者路径不对
- `500 Internal Server Error`：服务器内部出错

其中需要单独拎出来说明的两个：

- `401` 更像“你还没有通过身份校验”
- `403` 更像“你通过了身份校验，但你的身份没有对应的权限”

### 2. 响应头（Header）

响应头可以理解成：

- **服务器对这次响应的额外说明**

例如：

- 返回内容是什么格式
- 是否给客户端发了一份 Cookie
- 是否允许缓存

### 3. 响应体（Body）

响应体通常是服务器真正返回给你的数据内容。

很多接口返回的是 JSON，例如：

```json
{
  "id": 101,
  "title": "复习闭包",
  "isFinished": false
}
```

当前阶段可以先这么记忆：

- **JSON 常常出现在响应体里**
- **但 JSON 只是响应体的一种格式**

### 一个最小响应示意

```text
HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 101,
  "title": "复习闭包",
  "isFinished": false
}
```

这可以先读成：

- 服务器说这次请求在 HTTP 这层是成功的
- 返回内容类型是 JSON
- 具体数据放在响应体里

## 为什么接口经常返回 JSON

前面两章你已经看到 JSON 非常适合表达结构化数据：

- 有对象
- 有数组
- 有嵌套
- 能表达字符串、数字、布尔值、空值

所以当服务器要把一条任务、一组课程、一个用户资料返回给客户端时，JSON 就很适合作为承载格式。

因此在接口场景里，经常会形成这样一条链：

```text
客户端发 HTTP 请求
服务器返回 HTTP 响应
响应体里的数据格式是 JSON
客户端再把 JSON 解码成 Swift 类型
```

这里必须把边界看清：

- HTTP 解决的是“怎么通信”
- JSON 解决的是“数据长什么样”

## HTTP 为什么被称为“无状态”

这是后面理解登录态最关键的前置概念。

所谓 HTTP 是“无状态”的，当前阶段可以先近似理解成：

- **单看协议本身，服务器不会自动记得“上一条请求是谁发的”**

例如：

- 你刚请求了任务列表
- 下一秒又请求了个人资料

从协议角度看，这两次请求默认只是“两次独立请求”。

也就是说，HTTP 并不会天然替你保存：

- 这个用户刚刚已经登录过
- 这个请求和上一条其实来自同一个人
- 这个人上一次访问时已经通过校验

这也正是为什么“登录态”会成为一个专门问题。

## 没有状态时，登录态怎么保存

假设一个用户已经成功登录，后面客户端还要继续请求：

- 我的资料
- 我的任务列表
- 我的学习记录

服务器总得有一种办法判断：

- 这些后续请求是不是同一个已登录用户发来的

于是就会出现一类机制：

- 在登录成功后，给客户端一份后续可复用的凭证
- 客户端在后面的请求里，再把这份凭证带回来
- 服务器根据这份凭证判断身份是否有效

当前阶段最常见的两种概念就是：

- Cookie
- Token

先记住它们的共同点：

- **它们都和“后续请求如何延续身份”有关**

## Cookie 是什么

当前阶段可以先把 Cookie 理解成：

- **服务器发给客户端、并希望客户端在后续请求里继续带回来的一个小型状态凭证**

最典型的流程是：

1. 客户端发起登录请求
2. 服务器验证成功
3. 服务器在响应里通过 `Set-Cookie` 发下一份 Cookie
4. 客户端把这份 Cookie 保存起来
5. 后续再请求同一站点时，客户端把 Cookie 带回去
6. 服务器据此识别这是不是那个已经登录过的用户

你可以先把它想成：

```text
登录成功后
服务器：给你一张“后续来访时请带上的小票”
客户端：先保存
下次请求：把这张小票一起带上
```

一个最小示意如下：

```text
HTTP/1.1 200 OK
Set-Cookie: session_id=abc123
```

然后后续请求里可能会带：

```text
Cookie: session_id=abc123
```

当前阶段最重要的不是背字段细节，而是理解：

- Cookie 不等于用户本人
- Cookie 更像一个“与当前身份状态关联起来的标记”

## Token 是什么

当前阶段也可以先把 Token 理解成：

- **服务器签发或生成给客户端的一份访问凭证，客户端在后续请求里主动带上它**

它最常见的样子之一是放在请求头里：

```text
Authorization: Bearer <token>
```

你可以先把它理解成：

- 登录成功后，服务器给客户端一份令牌
- 客户端以后每次需要访问受保护资源时，都显式把这份令牌带上
- 服务器拿到令牌后，再判断它是否有效、是否过期、是否有权限

和 Cookie 一样，当前阶段最重要的是把角色看清：

- Token 不是用户资料本身
- Token 也不是权限本身
- 它是一份让服务器据此继续判断身份和访问资格的凭证

## Cookie 和 Token 最直观的区别

当前阶段只需要先抓住最直观的差异，不需要陷进实现流派争论。

### 共同点

- 都用于后续请求中的身份识别
- 都不是业务数据本身
- 都需要服务器配合校验

### 不同点

| 机制 | 携带方式 | 常见场景 |
| --- | --- | --- |
| Cookie | 浏览器自动携带 | 浏览器浏览网页 |
| Token | 调用方手动添加 | 程序调用接口 |

当前阶段不要急着问“哪种绝对更先进”。因为真实项目里，选择往往和：

- 系统形态
- 客户端类型
- 安全要求
- 服务端架构

都有关系。

对你来说，现在先把概念角色分清就够了。

## 登录态、权限、业务数据，不是一回事

这一点初学者很容易混在一起。

可以先把它们拆成三层：

1. **登录态**
   服务器能不能判断“你是谁”

2. **权限**
   服务器是否允许“你做这件事”

3. **业务数据**
   真正返回给你的任务列表、课程详情、用户资料

于是你就能理解为什么会出现下面这些情况：

- 没带 Token，服务器直接回 `401`
- 带了 Token，但你不是管理员，服务器回 `403`
- 带了合法 Token，服务器正常回 `200`，然后才在响应体里返回 JSON 数据

这三件事是分层发生的，不要混成“请求失败就是 JSON 有问题”。

## 网络失败、状态码错误、解码失败，是三层不同问题

这也是后面写客户端代码时必须有的意识。

### 1. 网络失败

例如：

- 设备没网
- 域名无法解析
- 请求超时

这时往往连一份正常 HTTP 响应都拿不到。

### 2. HTTP 状态不符合预期

例如：

- 拿到了响应，但状态码是 `401`
- 拿到了响应，但状态码是 `404`
- 拿到了响应，但状态码是 `500`

这时问题已经不是“有没有连上”，而是：

- 服务器明确回了一个不符合预期的结果

### 3. 解码失败

例如：

- 状态码是 `200`
- 服务器也确实返回了 JSON
- 但字段名、字段类型、结构层次和你的 Swift 模型对不上

这时问题又变成：

- 通信是通的
- HTTP 层也可能是成功的
- 但数据模型没对齐

所以后面一旦开始写 `URLSession`，你就应该逐步形成这种排查顺序：

1. 有没有拿到响应
2. 状态码是不是对的
3. 响应体是不是我预期的数据格式
4. JSON 能不能顺利解码成目标类型

## 读接口文档时，应该先看什么

后面真正对接接口时，不要一上来就只看返回 JSON 示例。

更稳妥的阅读顺序通常是：

1. **请求地址是什么**
   也就是 URL 或路径

2. **请求方法是什么**
   是 `GET`、`POST`，还是别的方法

3. **是否需要身份凭证**
   例如是否需要 Cookie，或者是否要带 `Authorization`

4. **请求参数放在哪里**
   是查询参数、路径参数，还是请求体

5. **成功时返回什么**
   状态码是什么，Body 长什么样

6. **失败时可能返回什么**
   例如 `401`、`403`、`404`、`500`

如果你只盯着 JSON 看，很容易漏掉真正决定请求能否成功的前置信息。

## 初学者最容易混淆的几个点

### 1. 以为 URL 和接口文档是一回事

不是。

- URL 只是资源地址
- 接口文档通常还会告诉你方法、参数、认证方式、返回格式、错误码

### 2. 以为拿到 JSON 就代表请求成功

不一定。

有些接口即使返回 JSON，也可能是错误信息 JSON。

也就是说：

- 返回了 JSON
- 不等于业务成功
- 不等于状态码一定是 `200`

### 3. 以为 HTTP 无状态，就表示服务器完全不保存任何东西

这也是常见误解。

HTTP 无状态更准确的意思是：

- 协议本身不会自动替你记住前一次请求上下文

但服务器完全可以：

- 自己保存会话信息
- 自己管理用户状态
- 再通过 Cookie 或 Token 让后续请求带回关联凭证

### 4. 以为 `401` 和 `403` 差不多

它们经常相关，但不一样。

- `401`：取决于能否判断“你是谁”
- `403`：取决于能否判断“你的权限是否足够”

## 本章小结

务必先把下面这几组边界记清：

- URL 解决“请求发到哪里”
- HTTP 解决“客户端和服务器怎样通信”
- JSON 解决“数据内容长什么样”
- Cookie 和 Token 解决“后续请求怎样继续表达身份”

同时也要记住后面排查网络请求时最重要的三层分界：

- 网络有没有通
- HTTP 响应是否符合预期
- JSON 是否能正确解码

如果这些边界清楚了，那么下一章再进入 `URLSession` 时，你看到的就不会只是一堆 API 名字，而会更像是在把：

- 请求地址
- 请求方法
- Header
- 状态码
- JSON 解码

重新拼成一条完整的数据流。
