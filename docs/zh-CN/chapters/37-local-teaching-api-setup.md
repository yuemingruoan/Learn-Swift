## 为什么这里不用第三方公开接口

如果教学示例直接依赖外部公开服务，通常会遇到这些问题：

- 服务端临时不可用
- 响应内容被对方调整
- 网络环境限制导致请求失败
- 演示、读者本地环境之间出现不一致

这些问题会让初学者陷入一种很麻烦的状态：

- 不知道自己是 Swift 代码写错了
- 还是外部接口刚好挂了
- 或者只是网络环境不稳定

因此从这一章开始，我们把目标收敛成：

- **后续网络请求默认都打到仓库自带的本地教学 API**

这样做的好处是：

- 数据结构可控
- 状态码可控
- 文稿、demo、练习和你本机看到的结果可以保持一致

## 这一章里的服务到底是什么

当前阶段你不需要把它理解成“我要学一门新的后端技术”。

你只需要把它理解成：

- 仓库里自带了一台很小的本地服务器
- 它会在你的电脑上监听一个本地端口
- 下一章里的 Swift 代码会请求它提供的固定 JSON

也就是说，这一章的重点不是“服务器怎么写”，而是：

- **怎样把后面要请求的接口先跑起来**

## 第一步：确认 Node.js 版本

进入终端后，先执行：

```bash
node --version
```

预期输出例如：

```text
v23.11.0
```

如果看到的主版本号是 `20` 或更高，例如：

```text
v20.18.0
```

说明环境已经满足要求。

如果终端提示找不到 `node`，或者版本低于 `20`，请先安装或升级 Node.js。

这一章后面会直接用到本地教学 API，因此这里最好一次把环境补齐，尤其是 macOS 用户。

### macOS：若未安装node，确认电脑中是否已安装 Homebrew

先执行：

```bash
brew --version
```

预期输出例如：

```text
Homebrew 5.1.1
```

如果你能看到版本号，说明 Homebrew 已经安装，可以直接跳到后面的“用 Homebrew 安装 Node.js”。

如果终端提示找不到 `brew`，说明还没有安装 Homebrew。

### macOS：安装 Homebrew

#### 方式一：中国大陆常用安装脚本

如果你在中国大陆网络环境下，通常更适合先用下面这条命令安装：

```bash
/bin/zsh -c "$(curl -fsSL https://gitee.com/cunkai/HomebrewCN/raw/master/Homebrew.sh)"
```

这样做的原因很现实：

- Homebrew 官方安装流程依赖 GitHub 等海外源
- 在中国大陆网络环境下，经常会因为网络限制而失败或极慢

所以如果你当前处在中国大陆，并且没有稳定的科学上网环境，优先使用这条命令通常更省事。

#### 方式二：Homebrew 官方安装脚本

如果你的网络可以正常访问官方源，例如你已经开启 VPN 或处在不受限制的网络环境下，也可以直接使用官方命令：

```bash
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
```

### macOS：安装完成后重新打开终端

Homebrew 安装完成后，最稳妥的做法是：

- 关闭当前终端窗口
- 再重新打开一个新的终端窗口

然后再次执行：

```bash
brew --version
```

如果现在已经能看到版本号，就说明 Homebrew 已经进入当前 shell 环境。

### macOS：用 Homebrew 安装 Node.js

确认 Homebrew 可用后，执行：

```bash
brew install node
```

安装完成后，再执行：

```bash
node --version
npm --version
```

你至少应该确认两件事：

- `node --version` 可以正常执行
- `Node.js` 主版本号不低于 `20`

例如：

```text
v20.18.0
10.8.2
```

这就说明后面启动教学 API 的运行时已经准备好了。

### 如果你不是 macOS 用户

本章主要面向 macOS 学习环境，因此这里把 Homebrew 路线写得最完整。

如果你使用的是其他系统，只需要完成同一个目标即可：

- 安装 Node.js
- 确保 `node --version` 可执行
- 确保主版本号不低于 `20`

## 第二步：进入教学服务目录

教学服务目录位于仓库根目录下：

```text
teaching-api/
```

进入该目录：

```bash
cd teaching-api
```

你会看到至少这些文件：

- `package.json`
- `server.mjs`

这里的 `package.json` 只是为了方便你通过 `npm run ...` 启动服务，本章并不需要你学习 npm 包管理细节。

## 第三步：启动教学 API

直接执行：

```bash
npm run start
```

默认情况下，服务会启动在：

```text
http://127.0.0.1:3456
```

当你在终端里看到类似输出时，就说明服务已经起来了：

```text
Learn Swift teaching API listening on http://127.0.0.1:3456
```

这里顺便补充一个基础概念：

- `127.0.0.1` 表示本机回环地址
- `3456` 是当前教学服务使用的端口

在后面的章节里，涉及到发送请求时默认都会从这个地址继续往下拼接路径：

```text
http://127.0.0.1:3456
```

## 第四步：先验证服务真的能访问

### 方式一：浏览器直接打开

你可以直接在浏览器访问：

```text
http://127.0.0.1:3456/health
```

如果启动成功，你应该会看到一个直观的 HTML 健康检查页面，页面上会明确告诉你：

- 本地教学 API 已成功启动
- 当前访问地址是什么
- 后续章节可以继续使用这台本地服务

### 方式二：用 curl 验证

如果你更习惯在终端确认，可以执行：

```bash
curl http://127.0.0.1:3456/health.json
```

如果返回的是一段 JSON，而不是连接失败，就说明服务已经可访问。

当前阶段不用提前记后续每一章会用到哪些接口。

更稳妥的学习顺序是：

- 这一章只确认服务已启动
- 到真正写请求的章节时，再解释当章需要的接口

## 如果启动失败，优先检查什么

### 1. 终端提示 `node: command not found`

这说明：

- Node.js 还没有安装好
- 或者没有加入当前终端环境

优先回到前面的版本检查步骤。

### 2. 终端提示端口已被占用

这说明：

- `3456` 端口已经被其他程序占用

当前阶段最简单的处理方式是：

- 先停止占用该端口的旧进程
- 或在启动时临时改用别的端口

例如：

```bash
PORT=4567 npm run start
```

但要注意：

- 如果你改了端口，下一章 Swift 示例里的 base URL 也要对应改掉

所以在学习阶段，更推荐保持默认端口。

### 3. 浏览器或 curl 提示连接失败

优先检查：

- 服务是不是还在运行
- 当前终端窗口有没有被你关闭
- 地址是不是写成了 `127.0.0.1:3456`
- 路径是不是写成了 `/health`

## 这一章真正完成的事情是什么

读到这里，你应该已经完成了下面这条准备链路：

1. 确认 Node.js 版本符合要求
2. 启动本地教学 API
3. 访问 `/health` 或 `/health.json` 验证服务可用

这意味着从下一章开始，你不再需要依赖外部第三方接口，而是可以把注意力集中到真正的学习重点上：

- `URLSession` 怎样发请求
- 请求真正发出去后，Swift 代码里会经过哪些步骤
- 响应拿回来后怎样检查状态码并解码

## 本章小结

务必先记住下面这几个事实：

- 后续网络章节默认依赖仓库自带的本地教学 API
- 这个服务端只需要你会启动和验证，不要求你现在学习后端细节
- 默认地址是 `http://127.0.0.1:3456`
- 浏览器健康检查页是 `/health`
- 命令行 JSON 检查地址是 `/health.json`

如果这些准备工作都完成了，那么下一章进入 `URLSession` 时，你看到的就不再是抽象概念，而是一条真正可以跑通的本地请求链路。
